Part I
Budget-Aware Reasoning:
What Already Exists
Three method families · worked examples · the gap
Part one. Let's survey what is already being done to control compute in LLM reasoning.

The Compute Cost Problem

Chain-of-thought reasoning works
Scaling thinking tokens improves accuracy on hard tasks — but linearly increases cost.
Most problems are easy
In practice, ~72% of MATH-500 problems need far fewer tokens than a worst-case budget allows (see histogram).
Fixed-budget policies waste compute
Setting one token limit for all queries: over-compute on easy ones, under-compute on hard ones.
MATH-500 · tokens used per solved problem (Qwen-2.5-Math-7B)
Central question
Can the model observe its own reasoning and decide — in real time — when to stop or retrieve?
Chain-of-thought and extended thinking work — but they are expensive. The key insight is that difficulty varies enormously across queries. A smart system should spend proportionally. Anthropic's own analysis shows 3-10x cost increase from extended thinking modes.

Three Families of Solutions

Family 1
Token-level throttling

  • Prune or skip generation steps
  • Inference-time, no retraining
  • Examples: TokenSkip, OPSDC
Signal: token probability / hidden-state entropy

Family 2
Reasoning-budget modeling

  • Model plans its compute allocation
  • Requires fine-tuning
  • Examples: SelfBudgeter, BudgetThinker, DEER, SeaKR
Signal: budget token / self-confidence

Family 3
Adaptive retrieval

  • Decide when to call retrieval tool
  • Mix retrieval & reasoning cost
  • Examples: FLARE, DRAGIN, Self-RAG, Adaptive-RAG
Signal: token probability / self-reflection
All three families share a common limitation: they operate on aggregate statistics (token count, perplexity, single scalar) rather than on the typed event structure of the reasoning chain.
I'll go through each family with a concrete example of how they work. The takeaway from all three is stated at the bottom: they all compress the reasoning process into a scalar signal and lose the structural information about what kind of step was just taken.

Family 1 · Token-Level Throttling

TokenSkip (Xia et al., 2025)
What: Train model to skip <think> segments selectively based on query complexity score.
How: Binary policy: for each reasoning segment, predict skip/keep from prompt embedding. Fine-tune with token-budget reward.
Limitation: Skip decision is made before the segment is generated — no observation of what actually happened inside the segment.
OPSDC (Yang et al., 2025)
What: Online pruning of speculative decoding candidates.
How: Draft model generates N candidate continuations; verifier scores them; low-value tokens pruned before acceptance.
Limitation: Operates purely at token level — no notion of reasoning step type (backtrack vs. conclusion).
Common signal: $$ s = \text{perplexity}(t_i) = -\log p(t_i | t_{<i}) $$ High perplexity → model is uncertain → candidate for skipping or pruning.
Key weakness: Same action (skip/prune) regardless of whether the model is in the middle of a backtracking loop, a final verification step, or the first exploration. The type of reasoning step is invisible.
TokenSkip is elegant: train once, run at inference time. But the skip decision happens before the step executes. OPSDC is about speculative decoding efficiency — orthogonal goal, but same structural problem. Both treat all tokens equally regardless of reasoning function.

Family 2 · Reasoning-Budget Modeling (Fine-tuning)

SelfBudgeter (Chen et al., 2025)
What: Model receives a token budget $B$ prepended to the prompt and learns to self-regulate within it.
How: SFT on traces with budget annotations. At test time: set $B$, model compresses reasoning accordingly.
Limitation: Budget is a single scalar set before the query. Cannot adapt when a problem turns out harder than expected mid-way.
BudgetThinker (Xie et al., 2025)
What: RL-trained model learns a "think short / think long" meta-policy.
How: GRPO reward combines accuracy + token efficiency. Two modes: quick (no extended thinking) and deliberate (full CoT).
Limitation: Binary mode switch; no fine-grained redistribution within a reasoning trace.
Shared architecture pattern:

Prompt: [budget=512] Solve: ...

Model is trained to honor the budget via SFT reward: $$ r = \mathbf{1}[\text{correct}] - \lambda \cdot \max(0, |T| - B) $$ where $|T|$ is tokens used and $B$ is the budget.
These methods work well on benchmarks with uniform difficulty. They struggle when the test distribution has high variance in problem difficulty, since the budget is fixed at query time.
SelfBudgeter and BudgetThinker are fine-tuned approaches. They are better than TokenSkip because the model can plan. But the budget is committed before the reasoning starts. There is no mechanism to say "I started a backtracking loop I didn't expect — let me stop early."

Family 2 · Confidence-Based Early Exit

DEER (Zhang et al., 2025)
What: Early exit from reasoning when model entropy drops below a threshold.
How: At each decoding step, compute $H(p) = -\sum_v p_v \log p_v$. If $H < \tau$ for $k$ consecutive steps, stop.
Limitation: Entropy threshold $\tau$ must be hand-tuned per task. No statistical guarantee on accuracy loss.
EntropyAT + Conformal (Ye et al., 2025)
What: Conformal prediction wrapper gives a coverage guarantee on top of entropy-based stopping.
How: Calibration set used to set $\tau$ such that $P(\text{correct} | \text{stop}) \geq 1 - \alpha$ with finite-sample guarantee.
Limitation: Guarantee is marginal over queries — not conditional on reasoning depth or event type.
Conformal calibration procedure:

Given calibration set $\mathcal{D}_{cal}$, find threshold: $$ \hat{\tau} = \inf\left\{\tau : \frac{|\{i: \text{correct}_i \text{ if stop at } \tau\}|}{|\mathcal{D}_{cal}|} \geq 1-\alpha\right\} $$ Finite-sample guarantee: $P(\text{error}) \leq \alpha + \frac{1}{|\mathcal{D}_{cal}|+1}$
Conformal approach is principled, but the stopping signal is still a scalar (entropy). Two queries with the same entropy but different reasoning structures (one mid-backtrack, one finalizing) receive the same treatment.
EntropyAT is the most principled approach in this family. It gives a real statistical guarantee. But the guarantee is marginal — it says "on average over the population of queries, you'll be correct 1-alpha of the time." It doesn't account for the fact that the model is in a backtracking loop right now.

Family 3 · Adaptive Retrieval — Predictive Triggering

FLARE (Jiang et al., 2023)
What: Retrieve when the model is about to generate uncertain tokens.
How: After generating a sentence, check if any token has $p < \theta$. If so, regenerate that sentence with retrieved context injected.
Limitation: Retrieval triggered by individual low-probability tokens — noisy signal that causes spurious retrievals on rare-but-correct tokens.
DRAGIN (Su et al., 2024)
What: Dynamic retrieval using attention entropy to identify which tokens need grounding.
How: Compute attention entropy $H_{\text{attn}}(t)$ for each token; high-entropy tokens indicate knowledge gap → form query from those tokens' context.
Limitation: Attention entropy correlates with genuine uncertainty but also with syntactic complexity and style variation.
DRAGIN attention entropy: $$ H_{\text{attn}}(t_i) = -\sum_j \alpha_{ij} \log \alpha_{ij} $$ where $\alpha_{ij}$ are attention weights from token $i$ to context token $j$.

Query formed from tokens $t_j$ with $H_{\text{attn}}(t_j) > \theta$: $$ q = \{t_j : H_{\text{attn}}(t_j) > \theta\} $$
Both FLARE and DRAGIN excel at deciding when to retrieve. They do not model the sequence of retrieve events — each decision is made independently.
FLARE and DRAGIN are focused on retrieval triggering specifically. DRAGIN is smarter than FLARE because it uses attention to identify what the model doesn't know, not just that it doesn't know. But both make each retrieval decision independently — no memory of "I already retrieved twice and each time returned low-relevance docs."

Family 3 · Adaptive Retrieval — Self-Evaluation

Self-RAG (Asai et al., 2023)
What: Model generates special reflection tokens to decide: retrieve? is retrieved passage relevant? is generated text grounded?
How: Five special tokens: [Retrieve], [IsRel], [IsSup], [IsUse], [No Retrieve]. SFT on annotated corpus. Beam search over reflection tokens at inference.
Limitation: Reflection tokens are auto-regressive ($r_t$ conditioned on $r_{<t}$), so prior retrievals are visible in context. However, Self-RAG cannot forecast cumulative retrieval cost $\Lambda_{\text{retrieval}}(T)$ or make explicit budget-aware decisions.
Adaptive-RAG (Jeong et al., 2024)
What: Classify query complexity into {no-retrieval, single-retrieval, multi-retrieval} before answering.
How: Small classifier (fine-tuned on query features) routes to appropriate pipeline.
Limitation: Routing decision is fixed at query start. Cannot switch to more retrievals if the initial single-retrieval answer is inadequate.
Self-RAG decoding: $$ \hat{y} = \arg\max_{y, r_1,\ldots,r_T} \sum_{t} \log p(y_t, r_t | x, y_{<t}, r_{<t}) $$ where $r_t \in \{\text{Retrieve, NoRetrieve}\}$ are reflection tokens predicted jointly with output tokens.
Adaptive-RAG routing: $$ \hat{c} = \text{Classifier}(x) \in \{\text{direct}, \text{single}, \text{multi}\} $$ Pipeline selected before any generation begins.
Self-RAG is the most sophisticated of the retrieval methods. It trains a meta-cognitive capability into the model. But the reflection tokens are emitted one at a time and each is independent of the history of prior retrievals. Adaptive-RAG makes a one-shot routing decision — if it's wrong, there's no recovery mechanism.

The Fundamental Gap

What existing methods see:

A sequence of tokens $t_1, t_2, \ldots, t_n$ with associated scalar signals:
perplexity, entropy, attention weight, budget counter

What is actually happening:

A typed event stream with causal structure:

backtrack at $t=12.3$s
verify at $t=14.1$s
retrieval-call at $t=15.0$s
evidence-merge at $t=15.9$s
commit at $t=17.2$s

The missing capability:

History-dependent rate prediction
After seeing 3 backtracks in 10 seconds, how likely is the next token to be another backtrack vs. a commit? Existing models cannot answer this.
Cross-event cost accounting
A retrieval call costs $c_r$ tokens. A backtrack loop costs $c_b$ tokens. Total expected spend is $\sum_k c_k \cdot \Lambda_k(T)$ — not just "tokens so far."
Type-conditional stopping
Early exit is safe after commit. Dangerous after backtrack. Same token count, different right answer.
This is the key insight motivating our work. All existing methods reduce the reasoning chain to a scalar. But reasoning has structure: there are backtracks, verifications, retrievals, commits. These are different events with different implications for future cost and accuracy. A model that can track this event structure can make much better decisions about when to stop or retrieve.

Prior Methods Are Special Cases of Hawkes

Claim: roughly half of the methods surveyed in Part I can be recovered by restricting a multivariate Hawkes process — different inductive biases, but one shared skeleton.

This is a unification claim: a multivariate Hawkes process with signed kernels and softplus link function is a strict generalization of at least five methods we surveyed. Part two. Now I'll explain our approach in detail.

Core Idea in One Picture

Observation: A reasoning chain emits a stream of typed events at irregular times.
subgoal-done backtrack verify retrieval-call commit
t=0 ────────────────────────── t=T
Model: Each event type $k$ has a conditional intensity (instantaneous rate): $$ \lambda_k(t \mid \mathcal{H}_t) = \mu_k + \sum_{j: t_j < t} \phi_{k,k_j}(t - t_j) $$ where $k_j$ is the type of the $j$-th event and $\mathcal{H}_t$ = history before $t$.
Budget control: Cumulative expected events through time $T$: $$ \Lambda_k(t, B_{\max}) = \int_0^t \lambda_k(s \mid \mathcal{H}_s)\, ds $$ Total cost $= \sum_k c_k \Lambda_k(T)$. Stop when $\sum_k c_k \Lambda_k(t) \geq B_{\max}$.
Why this is better
The intensity $\lambda_k(t)$ captures how past events causally increase or decrease future event rates. Three backtracks raises $\lambda_{\text{commit}}$ less than one backtrack followed by a verify.
The core innovation is modeling the reasoning chain as a marked temporal point process. Instead of looking at a scalar like perplexity, we track typed events and their occurrence times. The Hawkes process models how past events causally influence future event rates — this is the key capability missing from all existing approaches.

Temporal Point Processes — From Scratch

What is a point process?
A random sequence of event times $t_1 < t_2 < \ldots < t_N$ in $[0, T]$.
Characterized by its conditional intensity: $$ \lambda(t \mid \mathcal{H}_t) = \lim_{\Delta \to 0} \frac{P(\text{event in } [t, t+\Delta] \mid \mathcal{H}_t)}{\Delta} $$ This is the instantaneous rate given all past events.
What makes Hawkes special?
Hawkes process: each past event excites the rate for future events: $$ \lambda(t) = \mu + \sum_{t_i < t} \phi(t - t_i), \quad \phi(\tau) = \alpha e^{-\beta \tau} $$ Events cluster: one event makes more events likely. This matches reasoning: one backtrack triggers more backtracks.
Generalized signed multivariate Hawkes:
Each event has a type (mark) $k \in \{1,\ldots,K\}$.
Signed kernel matrix $[\phi_{kk'}]$ with $\alpha_{kk'} \in \mathbb{R}$:
$\phi_{kk'} > 0$ = excitation, $\phi_{kk'} < 0$ = inhibition. $$ \lambda_k(t) = \text{softplus}\!\left(\mu_k + \sum_{k'=1}^{K} \sum_{t_i^{k'} < t} \phi_{kk'}(t - t_i^{k'})\right) $$ Softplus ensures $\lambda_k(t) \geq 0$ even with negative kernels.
Why not RNN / Transformer?
Hawkes gives: (1) interpretable signed causal structure in $[\phi_{kk'}]$, (2) tractable intensity, (3) direct integration $\Lambda_k(t)$ for budget forecasting.
For those unfamiliar with point processes: a point process is a model for random event times. The conditional intensity lambda(t | H_t) is the instantaneous event rate given history. The Hawkes process adds mutual excitation: each event increases the rate of future events. The multivariate (marked) version has cross-excitation: a retrieval-call might increase the rate of evidence-merge. This is exactly the right model for reasoning chains where events causally influence each other.

Hawkes in Action: Intensity During LLM Reasoning

Two intensity curves respond to observed events — exactly the signal EventBudget uses to decide stop/continue.

λ_backtrack spikes after each backtrack and compounds — self-excitation. Stage A fires when λ_backtrack stays elevated and λ_commit is still flat: the model is looping, budget is wasted.
λ_commit rises only after subgoal-done events — cross-excitation signal that the model is converging. Stage B uses Λ_commit(t, B_max) to estimate how many more tokens are needed.
Same trace · what DEER-class methods see (scalar token entropy only):
DEER sees one noisy scalar and one threshold θ. The entropy profile inside the looping zone is indistinguishable from the profile inside the recovery zone — both trigger θ crossings. Result: false stops on legitimate verify/subgoal sequences, or missed stops on genuine loops. No amount of threshold tuning recovers the typed signal above.
Walk the audience through this left to right. First backtrack: lambda_backtrack spikes from the base rate mu. Second backtrack before lambda has fully decayed: the intensity compounds — self-excitation in action. In the Stage A zone, lambda_backtrack is elevated, lambda_commit is flat — EventBudget fires Stage A: the model is unlikely to solve within budget. After verify and subgoal-done, lambda_backtrack decays and lambda_commit rises sharply — cross-excitation from subgoal completion. No scalar threshold can distinguish "backtracking but will recover" from "unrecoverable loop". The Hawkes integral over full history does. The lower strip shows what a scalar method like DEER has access to on the same trace: the question marks are threshold crossings that the method cannot disambiguate. Same statistic, two completely different situations.

Preliminary Result · Hawkes > Poisson on MATH-500

Setup
Model: Qwen-2.5-Math-7B (reasoning mode).
Traces: MATH-500 CoT, $N \approx 1{,}500$ sessions.
Events: Level-1+2 detector (regex + hidden-state classifier), $K=6$ types.
Fit two models on held-out split: homogeneous Poisson $\lambda_k = \mu_k$ vs. multivariate Hawkes with exponential kernels.
Why this comparison? Poisson is the maximum-entropy baseline — it assumes independence of inter-event intervals. If Hawkes fits better, history matters: events are self- and cross-exciting, which is the founding assumption of EventBudget.

EventBudget Architecture

Hawkes Encoder
Input: event history $\mathcal{H}_t = \{(t_i, k_i)\}$
Outputs intensity vector $\boldsymbol{\lambda}(t) = [\lambda_1(t), \ldots, \lambda_K(t)]$

Architecture: event embedding → positional encoding (relative time) → 4-layer Transformer → intensity head with softplus activation. $$ \lambda_k(t) = \text{softplus}\left(W_k \cdot h_t + b_k\right) $$
Budget Forecaster
Numerical integration of $\Lambda_k(t) = \int_0^t \lambda_k(s)\,ds$
Computes total expected cost: $C(t) = \sum_k c_k \Lambda_k(t)$
Extrapolates $C(t + \Delta t)$ for lookahead planning.
Full model specification:

$d_{\text{model}} = 256$, 4 Transformer layers, 8 heads
$K = 6$ event types per domain
Event embedding: $\mathbb{R}^{K} \to \mathbb{R}^{d}$ (one-hot + time encoding)

Time encoding: $\text{TimeEnc}(\tau) = [\cos(\omega_1 \tau), \sin(\omega_1 \tau), \ldots]$
Trainable frequencies $\omega_i$.
The Hawkes encoder is trained jointly with the LLM policy via GRPO. The encoder's intensity outputs feed both Stage A (feasibility) and Stage B (retrieval redistribution).
The architecture has two main components. The Hawkes encoder takes the event history and outputs an intensity vector — the instantaneous rate of each event type. The budget forecaster integrates these intensities to predict total future cost. Both are trained jointly with the LLM policy.

Stage A · Feasibility Gate & Stage B · Budget Redistribution

Stage A — Early Exit / Feasibility
Evaluates after every reasoning event (continuous criterion, not a discrete trigger).

Computes cumulative forecast $\Lambda_{\text{commit}}(t, B_{\max})$: expected number of commit events before budget exhausted.

If $\Lambda_{\text{commit}}(t, B_{\max}) < \epsilon$ → problem is infeasible under current budget → exit early with best-so-far answer.

Calibrated via split conformal prediction: $\epsilon$ set so that $P(\text{error at exit}) \leq \alpha$ marginally over the calibration set.
Stage B — Retrieval Redistribution
Active when current intensity $\lambda_{\text{retrieval}}(t)$ is high but budget is nearly exhausted.

Computes: should we spend remaining budget on a retrieval call or on more reasoning?

$$ \text{Decision}(t) = \arg\max_{a \in \{\text{retrieve}, \text{reason}\}} \mathbb{E}[\text{reward} \mid a, \mathcal{H}_t, B_{\text{rem}}] $$
Correctness estimated via a lightweight surrogate $P(\text{correct} \mid a, \mathcal{H}_t, B_{\text{rem}})$ trained on labeled traces. Hawkes provides the event-sequence distribution $p(\text{future events} \mid a, \mathcal{H}_t)$ for sampling cost trajectories.
Key distinction from existing work: The decision uses the type and timing of past events, not just total token count. Stage A early exit after commit is very different from early exit after backtrack.
Stage A is the early exit mechanism. It asks: given what I've seen so far, is it realistic that I'll produce a correct answer before the budget runs out? Stage B is the retrieval decision mechanism. It asks: given remaining budget, should I retrieve or keep reasoning? Both decisions are made using the full event history, not just a scalar summary. Stage A uses conformal prediction for a statistical guarantee on the exit decision.

GRPO Training — Reward Function (Shao et al., 2024)

Reward decomposition: $$ R = r_{\text{acc}} + r_{\text{budget}} + r_{\text{early}} + r_{\text{format}} $$
$r_{\text{acc}}$ — Accuracy reward
Outcome: +1 if final answer correct, 0 otherwise.
For math: exact match after normalization.
For RAG: F1 against gold answer.
$r_{\text{budget}}$ — Token efficiency
$r_{\text{budget}} = \mathbf{1}[\text{correct}] \cdot \left(1 - \frac{C_{\text{used}}}{B_{\max}}\right)$
Zero if incorrect (no reward for cheap wrong answers).
$r_{\text{early}}$ — Early exit bonus
$r_{\text{early}} = \gamma \cdot \mathbf{1}[\text{Stage A exit}] \cdot \mathbf{1}[\text{correct}]$
Rewards correct early exits to incentivize Stage A calibration.
$\gamma = 0.3$ (hyperparameter).
$r_{\text{format}}$ — Event annotation
Small reward ($+0.05$) for correctly annotating reasoning steps with event tags. Encourages the model to maintain the event stream structure the encoder relies on.
Training: Phase 1 (SFT on gold traces) → Phase 2 (GRPO RL) → Conformal calibration on held-out val set.
GRPO is Group Relative Policy Optimization — similar to PPO but uses group-normalized rewards instead of a value function. The reward has four components. Accuracy is primary. Budget efficiency is only rewarded for correct answers — we don't want the model to be efficient and wrong. The early exit bonus and format reward are smaller but important for shaping the model's behavior toward the event-structured format we need.

DPO Alternative · Budget-Aware Preferences

Budget-conditioned DPO loss: $$\mathcal{L}_{\text{DPO-B}} = -\mathbb{E}_{(x, y_w, y_l, B)}\left[\log \sigma\!\left(\beta \Delta_{\pi/\pi_{\text{ref}}}\right)\right]$$ $$\Delta_{\pi/\pi_{\text{ref}}} = \log \tfrac{\pi(y_w | x, B)}{\pi_{\text{ref}}(y_w | x, B)} - \log \tfrac{\pi(y_l | x, B)}{\pi_{\text{ref}}(y_l | x, B)}$$ Budget $B$ is injected as a control token in the prompt; preferences are collected at fixed $B$.
Preference pair construction
P1 · Budget compliance: $y_w$ finishes within $B$, $y_l$ overruns.
P2 · Equal accuracy, shorter: both correct; $y_w$ uses fewer tokens.
P3 · Hawkes-informed: $y_w$ has lower $\int_0^T \sum_k c_k \lambda_k(t)\, dt$ — the pair is selected from traces where the intensity integral differs, not just token count.
Budget curriculum
Start with $B = B_{\max}$ (loose) → anneal to $B = 0.4 B_{\max}$ over training.
Monotone schedule; stronger selection pressure as model improves.
Rationale: easy-to-hard reduces reward hacking (policy can't collapse to short nonsense in phase 1).
DPO vs. GRPO — when to pick which
DPO: no explicit reward model; stable with sparse labels; budget enters via conditional preferences.
GRPO: group-relative advantage; better when reward is dense and decomposable ($r_{\text{acc}} + r_{\text{budget}} + r_{\text{early}} + r_{\text{format}}$).
Plan: DPO for Phase 1 (SFT warmup), GRPO for Phase 2 (fine-tuning with Hawkes signal).
Preferences sourced from 2-run pairs on same prompt: each question answered at two budgets $(B, B')$, Hawkes integral selects the winner.
This slide is new — it presents DPO as an alternative to GRPO for training the budget-aware policy. The key novelty is the Hawkes-informed preference pair (P3). Standard DPO would compare on accuracy or token count. We compare on the Hawkes intensity integral, which is a finer-grained signal: it captures whether the trace had lots of backtracking (wasted tokens) or not. The budget curriculum is borrowed from curriculum learning — starting with loose budgets prevents the model from collapsing to a trivial solution early in training. Decision: we'll try both DPO and GRPO. DPO is simpler to implement and more stable; GRPO has tighter theoretical connection to our reward formulation.

Theorem 1 — Convergence of Retrieval Spend

Theorem 1 (a.s. convergence).
Under the EventBudget policy $\pi^*$ trained with GRPO reward $R$, with Hawkes encoder satisfying Assumptions A1–A3 (stationarity, bounded excitation norm, ergodic reward process): $$ \frac{1}{N} \sum_{i=1}^{N} C_i^{\text{retrieval}} \xrightarrow{\text{a.s.}} C^* \leq \alpha B_{\max} $$ as $N \to \infty$, where $\alpha \in (0,1)$ is the target retrieval spend fraction.
Assumptions A1–A3
A1 (Stationarity): Hawkes intensity satisfies $\|\Phi\|_\infty < 1$ (spectral radius condition — prevents intensity explosion).

A2 (Bounded excitation): $\sup_t \lambda_k(t \mid \mathcal{H}_t) \leq M < \infty$ a.s.

A3 (Ergodicity): The Markov chain on event histories is geometrically ergodic.
Proof: (1) Stationarity of Hawkes under A1. (2) Under the stationary distribution (A3), GRPO optimization yields $\mathbb{E}[C^{\text{retrieval}}] \leq \alpha B_{\max}$ by reward design (penalized objective). (3) Birkhoff's ergodic theorem promotes this expectation to an a.s. time-average: $\frac{1}{N}\sum C_i^{\text{retrieval}} \xrightarrow{\text{a.s.}} \mathbb{E}[C^{\text{retrieval}}] \leq \alpha B_{\max}$.
Theorem 1 is our main theoretical result. It says that in the long run, the policy will converge to spending no more than alpha times the budget on retrieval calls. The three assumptions are standard for Hawkes processes. A1 is the stationarity condition — the spectral radius of the excitation matrix must be less than 1, otherwise the process can explode. A3 (ergodicity) is what allows us to apply Birkhoff's ergodic theorem to get the almost-sure convergence.

From Theory to Practice · Approximate Stationarity

The gap: Theorem 1 requires $\|\Phi\|_\infty < 1$ (strict stationarity). Real reasoning traces are only approximately stationary — phase shifts near the answer, warm-up at the start. What survives?
Relaxation 1 — $\varepsilon$-stationary bound
If $\|\Phi\|_\infty \leq 1 - \varepsilon$ only on a fraction $1 - \delta$ of the trace window: $$\mathbb{E}[C^{\text{retrieval}}] \leq \alpha B_{\max} \cdot (1 + C_1 \delta + C_2 / \varepsilon)$$ Degrades gracefully — the guarantee loosens but does not vanish.
Relaxation 2 — Piecewise stationarity
Split the trace into quasi-stationary phases (change-point detection on intensity). Apply Theorem 1 per phase, union-bound aggregate.
Black-box Hawkes validator (proposed tool).
Input: any LLM $M$, domain $D$.
  1. Run $n$ probe queries from $D$ on $M$; extract events.
  2. Fit Hawkes, compute $\widehat{\|\Phi\|}_\infty$ with bootstrap CI.
  3. KS residual test: is point process well-specified?
  4. Report: fit score $\in [0, 1]$ + decision threshold.
Output: "EventBudget applicable to $M$ on $D$: yes / caution / no."
Complements theoretical guarantees: even when A1 cannot be proven, the validator gives an empirical pass/fail per model.
This slide addresses a concrete question from the meeting: what do we do when stationarity doesn't hold? Two directions. First, look for a weakened theorem — an approximately-stationary bound with graceful degradation. If spectral norm is epsilon-close to the boundary, the retrieval-spend bound loosens by a factor proportional to epsilon. This is in the spirit of perturbation bounds for Markov chains. Second, a practical black-box validator: for any new LLM plus domain, run probe queries, fit Hawkes, check goodness-of-fit, and output a go/no-go decision. This is a deliverable by itself — it generalizes the framework beyond our specific training setup. This addresses Dmitry's concern: we don't have to abandon the method when conditions aren't met; we have falsifiable probes.

Bonus Probe · Are the "Let Me Reconsider" Tokens Real?

Concern. RL-tuned reasoners emit verbal markers ("hmm, let me reconsider", "wait, actually…") that may be cosmetic — learned to pad long CoT, not to signal a real state shift.
Chinese-room failure mode: raise-right-hand tokens with no internal correlate.
Hawkes as an authenticity probe
Align textual markers $m_j$ with detected events.
Compute lead-lag intensity: $$\Delta\lambda_k(\delta) = \lambda_k(t_{m_j} + \delta) - \lambda_k(t_{m_j} - \delta)$$ for $\delta \in [50, 200]$ tokens.
Authentic reasoning
$\lambda_{\text{backtrack}}$ spikes before the verbal marker — intensity leads language. The text reports what the model is already doing.
Cosmetic marker
$\lambda_{\text{backtrack}}$ flat before marker, rises after — the marker is a trigger, not a report. Token habit, not state shift.
Side product: EventBudget becomes an interpretability diagnostic — it can flag pre-training corpora or RL-tuned models where reasoning tokens lack internal support.
This addresses the interpretability concern raised in discussion: do the surface tokens actually correspond to internal reasoning, or are they learned padding? Our method operates on surface events only — but we can turn that into a feature. By comparing textual markers to intensity dynamics, we can distinguish reasoning that leads language (authentic) from language that triggers reasoning-shaped tokens (cosmetic). Practical value: this is a diagnostic tool for the broader LLM community. It's a side product of EventBudget, not the main contribution, but it's cheap to add since we already have both the event stream and the textual markers. This connects to Anthropic's interpretability research direction — without actually needing weight access.

Why the Dataset is the Hard Part WIP

Problem 1: No existing TPP reasoning dataset
Published datasets have final answers, sometimes CoT text. None have time-stamped, typed reasoning events. We must create this from scratch.
Problem 2: Event detection is hard
Regex: fast but noisy (30% false positive rate on early tests).
Classifier on hidden states: better but requires labeled data.
LLM judge: accurate but slow (0.5s/trace on GPU).
→ Need a three-level funnel.
Problem 3: Labels require running models at scale
55K traces × Qwen-7B × extended thinking = significant compute.

We aim to use open-weight models (Qwen-2.5) to reduce cost, at the price of lower annotation quality.
This is why Part III is marked WIP: the pipeline design is complete, but the actual data generation has not started.
The dataset is the critical dependency for everything else. You can't train the Hawkes encoder without event sequences. Four-Stage Pipeline Overview WIP

Stage 1
Trace collection

Run Qwen-2.5-Math-7B-Instruct on MATH-500 + AIME with extended thinking.

55K traces, each with full CoT text + final answer.

Filter: keep traces >200 tokens.

Stage 2
Event detection

Three-level funnel:
1. Regex patterns (fast, noisy)
2. Binary classifier on hidden states
3. LLM judge (gold set only)

Output: event stream $(t_i, k_i)$ per trace.

Stage 3
Stage A/B labels

For each trace: find first correct commit.
Label all events before it as "pre-commit."

Stage A label: is problem feasible at each event?
Stage B label: would retrieval here have helped?

Stage 4
Dataset packaging

HawkesTrainSample dataclass:
event_times, event_types, budget_used, stage_a_label, stage_b_label, answer_correct.

Train/val/test: 80/10/10%.
The four stages are sequential and each depends on the previous. Stage 1 is straightforward — run the model and save traces. Stage 2 is the hard engineering work — building and evaluating the three-level event detector. Stage 3 requires the "oracle" question: at what point was the problem solvable? Stage 4 packages everything into a format the Hawkes encoder can train on.

Stage 2 · Three-Level Event Detector WIP

Level 1 — Regex (speed: <1ms/trace)
Pattern matching on CoT text segments.
backtrack: /wait|actually|let me reconsider/i
verify: /let me check|verify:|double.check/i
commit: /the answer is|\\boxed\{/i
Recall ≈95%, Precision ≈70%. Used for initial filtering.
Level 2 — Classifier (speed: ~10ms/trace)
Logistic regression on 256-dim hidden state from LLM at segment boundary.
Trained on Level 3 gold labels.
Expected F1 ≈87% based on similar approaches in Self-RAG.
Level 3 — LLM Judge (speed: ~500ms/trace)
Prompt to Qwen-2.5-72B-Instruct:
"Given this reasoning segment, classify the cognitive action: [backtrack / verify / branch / subgoal-done / failed-attempt / commit]"
Applied to a random 5% sample → gold set of ~2750 traces.
Funnel logic:
All traces → Level 1 → Level 2 for regex-flagged events → Level 3 for disagreements. Final detector: Level 2 predictions, calibrated against Level 3 gold labels.
The three-level funnel is designed to balance speed and accuracy. Level 1 (regex) is applied to everything — fast but noisy. Level 2 (hidden state classifier) is applied to the events flagged by Level 1 — better accuracy. Level 3 (LLM judge) is applied only to a sample to create a gold set for calibrating Level 2. This is similar to how Self-RAG built its reflection token annotations, but cheaper because we use a smaller judge model.

Stage 3 · Stage A / B Labeling Algorithm WIP

find_first_correct_commit algorithm: 
def find_first_correct_commit(trace, oracle):
    events = trace.events
    for i, event in enumerate(events):
        if event.type == "commit":
            answer = extract_answer(
                trace.text[:event.char_pos]
            )
            if oracle.is_correct(answer, trace.gold):
                return i  # first correct commit index
    return None  # no correct commit found
Stage A label
For each event $e_i$ in trace, Stage A label = 1 if find_first_correct_commit returns an index $j > i$ with $C(t_j) \leq B_{\max}$.
i.e., "at this point in the trace, the problem is still solvable within budget."
Stage B label
For each retrieval-call event, Stage B label = 1 if the next answer-commit after this retrieval is correct, AND removing this retrieval would have made the commit incorrect.
i.e., "this retrieval was load-bearing."
Stage A labels: run oracle forward from each event — is a correct commit reachable before $B_{\max}$? Stage B labels require counterfactual evaluation: run model with vs. without each retrieval call. Both require model re-execution — the main compute bottleneck.
The labeling algorithm is straightforward but computationally expensive. For Stage A: at each event, does the problem remain solvable within budget? This requires knowing the future — we use the gold answer as oracle. For Stage B: was this retrieval actually helpful? This requires counterfactual evaluation — what would have happened without the retrieval? The counterfactual evaluation means running the model twice per retrieval event per trace.

Open Engineering Challenges WIP

Challenge 1
Timestamp alignment
Token timestamps are not naturally available from batch inference. Must record per-token generation time, which requires single-stream inference (no batching) or careful profiling.
Challenge 2
Event detector calibration
Level 2 classifier trained on Level 3 gold labels. Gold set is 5% sample — may not cover all event types evenly. Need stratified sampling to ensure rare events (e.g., failed-attempt) are in the gold set.
Challenge 3
Counterfactual evaluation
Stage B labels require counterfactual: "what if this retrieval hadn't happened?" Running model from each retrieval point is expensive. Exploring use of cached KV states to reduce cost.
Challenge 4
Domain R traces
Multi-hop RAG reasoning traces require a retrieval-augmented model. Need to instrument a RAG system to log retrieval calls as TPP events. HotpotQA with Contriever retriever is the planned setup.
Challenge 5
Hawkes stationarity
Need to verify Assumption A1 ($\|\Phi\|_\infty < 1$) on trained encoder. If violated, intensity can diverge on long traces. Plan: regularize excitation matrix norm during training.
Challenge 6
Training stability
GRPO is known to be unstable with sparse rewards (correct/incorrect). Plan to use dense shaping rewards (format + event annotation) to stabilize early training.
Six open challenges. The first is subtle: to fit a temporal point process, we need actual timestamps, not just event order. Batch inference doesn't give us per-token times. Challenge 3 (counterfactual) is the most expensive: running the model from each retrieval point. Challenge 5 is a theoretical requirement: if the stationarity assumption is violated, the Hawkes process can produce infinite intensity — a catastrophic failure mode.
